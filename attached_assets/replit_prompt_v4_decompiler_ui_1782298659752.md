## Part A: Ghidra decompiler integration (pseudocode)

### What "full integration" means here, precisely
Add a "Pseudocode" tab/view that, for whichever function is currently
selected in the function list, shows actual decompiled C-like pseudocode
generated by Ghidra's standalone C++ decompiler — not a placeholder, not a
disassembly-to-pseudo-C text transform written by hand, not a stub that
returns the function's existing disassembly reformatted to look like C. If
real decompilation cannot be wired up working in this round, STOP and report
back exactly what's blocking it rather than shipping a fake/placeholder
pseudocode view that looks done but isn't.

### Components to integrate (Apache 2.0 licensed, from Ghidra's GitHub repo)
1. **SLEIGH runtime + AArch64 spec**: pulls from
   `Ghidra/Features/Decompiler/src/decompile/cpp/sleigh*.cc/.hh` (the SLEIGH
   interpreter) and `Ghidra/Processors/AARCH64/data/languages/` (the
   `.slaspec`/`.sinc` files defining ARM64 → p-code lifting).
2. **The SLEIGH compiler** (`slgh_compile.cc` and related): build this as a
   host-side tool (runs on the CI machine / build machine, NOT on Android) to
   compile `AARCH64.slaspec` into a binary `AARCH64.sla` file ONCE. Ship the
   resulting `.sla` as an asset bundled in the APK (e.g.
   `app/src/main/assets/AARCH64.sla`) — do not attempt to run the SLEIGH
   compiler on-device.
3. **The C++ decompiler core**: the rest of
   `Ghidra/Features/Decompiler/src/decompile/cpp/` (funcdata.cc, action.cc,
   type.cc, print*.cc, etc.) — this runs ON-DEVICE, cross-compiled for
   `arm64-v8a` via the NDK alongside the existing Capstone/sqlite3 native
   code. Add it as a new CMake target/module in `app/src/main/cpp/`, follow
   the same pattern already used for the existing native code in
   `CMakeLists.txt`.

### Integration driver (new C++ code you write)
Write a new file, e.g. `app/src/main/cpp/decompiler_bridge.cpp`, that:
1. Takes a function's address + byte range (already available from the
   existing `functions` table / ELF parser data)
2. Loads the bundled `AARCH64.sla` and feeds the function's raw bytes through
   the SLEIGH runtime to get p-code for that function
3. Constructs the minimal program-model object the decompiler core expects
   (`Architecture`, a `loadimage` populated with just this function's bytes
   at the correct address, `Funcdata`) — do not attempt whole-binary analysis
   up front; scope this to one function at a time, matching the per-function
   streaming approach already decided for memory reasons
4. Runs the decompiler's action pipeline on that `Funcdata` and calls its
   pretty-printer to get a pseudocode string
5. Returns that string back across JNI to Kotlin

Expose this via JNI as something like:
`PeekNative.decompileFunction(handle: Long, address: Long): String`

### Performance/memory behavior
- Decompile on-demand, lazily, only when the user opens the Pseudocode tab
  for a given function — do not eagerly decompile every function in the
  binary on load.
- Cache decompiled output in the existing SQLite cache (add a
  `pseudocode` column to `functions`, or a new linked table) so re-opening
  the same function doesn't re-run the decompiler.
- Free all SLEIGH/decompiler working memory after each function completes —
  don't hold per-function decompiler state around once the string result is
  captured.
- Run decompilation on a background thread/coroutine, never on the UI
  thread — show a loading indicator in the Pseudocode tab while waiting.

### REQUIRED validation before calling this done
Before reporting this round complete:
1. Pick a real function from a real test `.so` (e.g. `JNI_OnLoad` from
   `libSMGP.so`, or any function with a recognizable prologue/body already
   visible in the ASM tab).
2. Open its Pseudocode tab and confirm the output is genuinely structured
   C-like code (variable declarations, if/loop structures, recognizable
   operations matching what the ASM tab shows) — not raw disassembly
   reformatted, not an error message swallowed into an empty/placeholder
   string, not obviously wrong/garbage output.
3. If this validation fails or can't be completed, report exactly which step
   (SLEIGH lift, p-code generation, decompiler action pipeline, pretty-print)
   is failing and why, rather than papering over it.

## Part B: Header "view options" control

### Behavior
Add a compact icon-only button docked to the right side of the toolbar
(loaded-file-name header) — no text label, no emoji, use a vector drawable
(SVG converted to Android `VectorDrawable` XML, consistent with how
`ic_peek_logo.xml` etc. are already done as drawables in this project — check
`app/src/main/res/drawable/` for the existing icon style/format to match).

Tapping it opens a small popup/dropdown (e.g. `PopupMenu` or a custom popup
window — whichever fits better with checkable items; a `PopupMenu` with
checkable menu items is the simplest fit) listing every available
tab/view: ASM, Pseudocode, Hex, Exports, Imports — each with a checkbox
reflecting whether it's currently shown in the tab bar.

Toggling a checkbox immediately adds/removes that tab from the tab bar
(`TabLayout` + `ViewPager2` adapter) without needing to close/reopen the
popup or restart the screen.

### Defaults and persistence
- Default visible tabs, in this order: **ASM, Pseudocode, Imports, Exports**
- Hex is available in the menu but OFF by default — user opts in
- Persist the user's chosen set + order using `SharedPreferences` (simple
  key like a stored ordered list of tab IDs) so it's remembered across app
  restarts, not just for the current session
- If a user has zero tabs selected, don't allow the last one to be
  unchecked — keep at minimum one tab visible (ASM is the sensible
  always-available fallback) to avoid an empty content panel with no way
  back in

## Part C: Function list coloring (with detection)

### Detection logic (new C++ logic, e.g. in a new
`app/src/main/cpp/thunk_detector.cpp` or folded into existing analysis pass)

Classify each function into one of these categories using data already
available from the ELF parser + xref pass:
- **Imported/library thunk**: function body is small (e.g. under ~5
  instructions as a starting heuristic — tune if it produces obviously wrong
  results on test binaries) AND it consists essentially of a load-from-GOT +
  branch pattern (the `adrp`/`ldr`/`add`/`br` sequence already visible in the
  app's own ASM tab screenshots is exactly this pattern) AND/OR it directly
  targets a PLT-style import symbol (using the import detection fixed in the
  previous round). Mark these as **thunks**.
- **Has real symbol name** (from `.dynsym`/`.symtab`, not auto-generated
  `sub_`): mark as **named**.
- **Auto-discovered, no symbol** (`sub_<addr>` functions from the branch-
  target discovery pass added previously): mark as **local/unnamed**.

Store this classification (e.g. an enum/int column: `THUNK`, `NAMED`,
`LOCAL`) in the `functions` table, computed during the same analysis pass
that already builds the function list — don't add a separate slow second
pass if it can be folded into existing work.

### Visual treatment in the function list
Using the existing dark theme palette in `colors.xml` (add new color
entries if needed, but keep them tonally consistent — don't introduce a
jarring bright color that clashes with the existing palette):
- **Thunks**: a distinct accent color (this is the "pink in IDA" effect the
  user is referencing — pick something in that family, e.g. a muted
  magenta/pink, that reads clearly against the dark background without being
  garish)
- **Named functions**: normal/bold text (these are the "interesting,
  human-labeled" functions — bold or a slightly brighter text color to make
  them stand out from auto-generated ones)
- **Local/unnamed (`sub_`) functions**: current default styling, or slightly
  dimmed/secondary text color, since these are the most numerous and least
  individually meaningful

Apply this in `FunctionCompactAdapter` (and the full `FunctionAdapter` used
elsewhere if both are still in use) — check both adapters for consistency so
coloring doesn't only show up in one of the two function-list UIs in the app.

This coloring is independent of and should compose correctly with the
selected-function highlight from the previous round (selection background +
classification text color/weight should both be visible at once, not
conflict).
