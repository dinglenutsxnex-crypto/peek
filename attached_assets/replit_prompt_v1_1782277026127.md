# Project: ARM64 .so Analyzer for Android (v1 — disassembly + SQLite only)

## What this is
An Android app for static analysis of ARM64 (.so) shared library files, inspired by
IDA Pro's listing view. This v1 has NO decompiler and NO Python scripting — those are
explicitly out of scope for this iteration. Do not add them, stub them, or scaffold
for them. Keep the scope tight.

## Starting point
Fork this existing open-source project as the base:
https://github.com/yhs0602/Android-Disassembler

It already provides, in Kotlin/Java + a C++/JNI native layer:
- ELF file parsing (via ELFIO)
- ARM64 disassembly (via Capstone, already vendored as a native module)
- A working listing view UI (offset / bytes / mnemonic / operands columns)
- Branch-target highlighting and "follow jump" navigation
- Export/import symbol table tabs
- A proven Gradle + NDK + CMake build pipeline for native modules

Use this as the actual codebase to build on, not as a reference to reimplement.
Preserve its existing native module structure (it has `capstone`/`capstone2` native
module directories already wired into Gradle/CMake) — add new functionality
following that same pattern rather than restructuring the project.

## Core requirement: most logic must live in C++, compiled to .so
Disassembly, ELF parsing, and SQLite read/write should happen in C++ native code
(new or existing native modules), exposed to Kotlin via JNI. Kotlin/Java should be
a thin UI/orchestration layer only — it calls into native functions and renders
what comes back. This is a deliberate performance requirement (target device: low-end
Android, 4GB RAM, ~2GB free), not a style preference. Do not move parsing or
disassembly logic into Kotlin for convenience.

## What to build in this iteration

### 1. SQLite-backed analysis cache
When a .so file is opened and disassembled, persist the results to a local SQLite
database (use SQLite directly from C++ via the sqlite3 C API — no Room, no ORM)
so re-opening the same file loads instantly from cache instead of re-disassembling.

Minimum schema (adjust as needed, but cover this data):
- `functions` table: address, size, name (if symbol exists), end_address
- `instructions` table: address, size, bytes (hex), mnemonic, operands, function_id (FK)
- `symbols` table: address, name, type (import/export), is_import (bool)
- `xrefs` table: from_address, to_address, ref_type (call/jump/data)
- `binaries` table: file_hash (sha256 of the .so, used as cache key), file_path,
  last_analyzed_timestamp

Use the file's SHA256 hash as the lookup key, not the file path, so renamed/moved
copies of the same binary still hit the cache.

### 2. Basic xref computation
While disassembling, detect and record cross-references:
- Direct calls/branches (BL, B, B.cond, CBZ, CBNZ, etc.) → code-to-code xrefs
- ADRP+LDR / ADRP+ADD patterns referencing data → code-to-data xrefs
Store these in the `xrefs` table. Surface them minimally in the UI — e.g. when
viewing a function, show "called from: [list of addresses]" if data exists.
This does not need to be exhaustive or perfectly accurate in v1 — basic direct
xref detection only. Skip indirect/computed branch resolution for now.

### 3. UI additions to the existing listing view
- A way to see "called by" / "calls to" for the function currently in view,
  pulling from the xrefs table
- A simple function list view (address + name, sourced from the `functions` table)
  that lets the user jump to a function's disassembly
- Keep using the existing app's visual style/theme — don't redesign the UI,
  extend it

## Explicitly OUT of scope for this iteration — do not implement or scaffold:
- Decompiler / pseudocode generation (no Ghidra, no SLEIGH, no p-code)
- Python scripting or any scripting engine (no Chaquopy yet)
- Code/pseudocode dump-to-zip export feature
- Indirect/computed branch xref resolution
- Any cloud/server component — everything is fully on-device

## Build & CI requirement
The app must build via GitHub Actions, not just locally in Replit. Set up a GitHub
Actions workflow (`.github/workflows/build.yml` or similar) that:
- Checks out the repo and submodules (the base project uses git submodules —
  run `git submodule update --init --recursive`)
- Sets up JDK + Android SDK + NDK (use a maintained action like
  `android-actions/setup-android` or equivalent, pin specific NDK version)
- Runs the Gradle build (`./gradlew assembleDebug` at minimum; release build
  config can come later)
- Uploads the resulting APK as a build artifact

Make sure the native C++/CMake build (Capstone + the new SQLite-backed module)
is actually invoked as part of this Gradle build, not skipped — verify the .so
files are present in the final APK.

## Target constraints to keep in mind throughout
- Target low-end device: 4GB RAM (~2GB free at runtime), 64GB storage (~10GB free)
- Do not load an entire large binary's full disassembly into memory at once if
  avoidable — query SQLite for the visible range/function as needed rather than
  holding everything in a big in-memory structure
- This constraint matters more as binaries get larger; for v1 correctness over
  extreme optimization is fine, but don't design something that obviously can't
  scale later (e.g. don't hold full instruction lists in Kotlin-side memory)

## Deliverable
A working fork that: opens a .so file, disassembles it via the existing
Capstone pipeline, stores results + basic xrefs in SQLite via a new C++ native
module, displays the existing listing view plus a function list and basic
xref info, and builds a debug APK successfully via GitHub Actions.
